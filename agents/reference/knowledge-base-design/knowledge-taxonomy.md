# 四类知识框架与三级可信度体系

> 来源：刘宇帅《AI Coding 真正的护城河是知识库》
> 适配：ai-workflow-control-kit 的技能上下文设计

## 为什么理解知识分类对 Kit 重要

ai-workflow-control-kit 的 skills 本质上是"AI Agent 的知识执行封装"。理解知识的不同类型和可信度等级，能帮助你设计更精确的 skill：
- **加载策略**：什么知识常驻上下文，什么知识按需加载
- **约束级别**：什么规则可自动执行，什么规则需要人确认
- **故障预防**：什么情况下 AI 可能误判，如何设置护栏

## 四类知识

### 第一类：事实（Facts）

| 属性 | 说明 |
|------|------|
| **内容** | 代码可扫描的客观事实 — 接口签名、表结构、调用关系、MQ 事件、基线用例 |
| **来源** | 自动化扫描工具，为每种语言/框架设计独立扫描模板 |
| **可信度** | 最高（机器可验证，零脑补） |
| **核心原则** | **能扫绝不写** |
| **Kit 加载策略** | 按需检索，通过 references/ 引用 |

### 第二类：知识（Knowledge）

| 属性 | 说明 |
|------|------|
| **内容** | 代码看不出的业务理解 — 规则设计原因、接口的坑、历史决策 |
| **来源** | 两条人工通路：**主动补全**（不成文的规矩）+ **纠错反哺**（AI 犯错后记录） |
| **可信度** | 中等（取决于维护质量） |
| **核心原则** | 回答"为什么"，而非"是什么" |
| **Kit 加载策略** | 常驻上下文（SKILL.md 的 `## Context` / `## Constraints`），或按需加载 |

### 第三类：模式（Patterns）

| 属性 | 说明 |
|------|------|
| **内容** | 跨服务/跨技术域的组织级共识规范：语言规范、通用规范（DB/日志/错误处理/API）、中间件规范、测试规范、平台规范 |
| **来源** | 团队历史规范沉淀 + AI 汇总整理 + 持续维护 |
| **核心原则** | 保证 AI 产出风格一致 |
| **Kit 加载策略** | 宿主级配置（`claude/`、`codex/` 的 rules/）或技能级引用（`references/`） |

### 第四类：视图（Views）

| 属性 | 说明 |
|------|------|
| **内容** | 前三类的聚合重组，不原创内容；两个维度：**业务域维度**（按域聚合）、**业务系统维度**（按系统梳理） |
| **来源** | 聚合不重写 |
| **核心原则** | AI 一眼看到全局，不用满世界拼图 |
| **Kit 加载策略** | 可通过 `workflow-router` 编排组合多个技能形成视图 |

## 三级可信度体系

知识可信度决定了 AI 在 skill 执行中如何使用这些知识。

| 档位 | 含义 | 消费规则 | skill 中的使用场景 |
|------|------|---------|-------------------|
| **确定** | 直接来自代码，或多来源交叉验证一致 | 技术方案只吃确定档 | `Constraints` 中的硬性约束；`Success Criteria` 中的验收条件 |
| **推断** | 单一来源或带猜测成分 | 标"待确认"，仅供参考 | `Common Failure Modes` 中标记 ⚠️ 的条目；需人工审核的步骤 |
| **冲突** | 多来源对不上 | **最值钱的风险入口** | 专挑冲突做测试设计；出方案时标注"待人工确认" |

### 在 replay-autopilot 中的应用

replay-autopilot 的 oracle 对比可直接复用可信度体系：
- **确定档**的 oracle 断言设为强制通过
- **推断档**的 oracle 断言设为"参考但不强制"
- **冲突档**作为 test charter 的种子输入

## 与系统蒸馏方法论的对应关系

| 四类知识 | 蒸馏三层 | 说明 |
|---------|---------|------|
| 事实 | 明细层（reference/） | 可扫描的客观事实，按需检索 |
| 知识 | 语义层（domain/） | 业务意图和规则，常驻上下文 |
| 模式 | 能力层 + 规则标注 | 执行规范和约束 |
| 视图 | api/_index.md 编排 | 聚合前三类的业务视图 |

## 在 Kit 中的落地示例

以 `dev-workflow` skill 为例：

```yaml
# 在 SKILL.md 的 metadata 或 loading 策略中声明

knowledge_loading:
  # 常驻加载（语义层 + 模式层）
  always:
    - context.md                    # 知识类：业务上下文、"为什么"
    - behavioral-rules.md           # 模式类：编码规范约定

  # 按需加载（事实层 + 明细层）
  on_demand:
    - references/schemas/           # 事实类：接口签名、表结构
    - references/enums.md           # 事实类：枚举定义
    - references/error-codes.md     # 事实类：错误码
```
