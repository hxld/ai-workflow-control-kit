# 知识库三循环运营模型

> 来源：刘宇帅《AI Coding 真正的护城河是知识库》
> 适配：ai-workflow-control-kit 的 skill 持续改进

## 三循环总览

知识库最怕变成一潭死水。需要三个循环持续运转：

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  扫描循环     │ ──→  │  消费循环     │ ──→  │  反哺循环     │
│  (上行)       │      │              │      │  (下行)      │
│  代码→知识    │      │  知识→方案   │      │  纠正→知识   │
└──────────────┘      └──────────────┘      └──────────────┘
       ↑                                       │
       └───────────────────────────────────────┘
```

**只有上行没有下行，知识库就是个会过时的快照仓库**，和吃灰的文档库没区别。

---

## 循环 1：扫描循环（上行）

**代码 → 知识**：代码变更时自动刷新事实层。

在 Kit 中的实现：

| Kit 组件 | 用途 |
|---------|------|
| `hooks/` 的 `FileChanged` hook | 检测代码变更，触发知识点刷新 |
| `agents/skills/knowledge-refresh/` | 按需刷新特定领域的知识索引 |
| `replay-autopilot` 的定时轮询 | 周期性验证知识点是否过期 |

**实践建议**：
- 扫描循环只更新"事实"类知识（接口签名、表结构等）
- "知识"类（业务意图）不通过扫描更新，避免机器覆盖人工理解
- 扫描循环是自动的，但每次扫描结果需登录「推断」档，人工审核后升为「确定」

---

## 循环 2：消费循环

**知识 → 方案**：AI 执行工作流时从知识库取知识。

在 Kit 中的实现：

| Kit 组件 | 用途 |
|---------|------|
| `agents/skills/dev-workflow/` | 方案生成时加载领域知识 |
| `agents/skills/gen-tests/` | 生成测试时加载可信度"确定"档的知识作为基线 |
| `agents/skills/pre-flight-check/` | 风险检查时加载约束条件 |
| `workflow-router` | 跨技能的知识路由和编排 |

**消费规则**：
- 出方案只吃「确定」档 — 方案不能建在猜测上
- 「推断」档知识标注"待确认"后作为参考
- 「冲突」档是风险入口，方案中标"待人工确认"

---

## 循环 3：反哺循环（下行）⭐ 最重要

**纠正 → 知识**：AI 犯错、人纠正后回写进知识库。

这是三个循环中最核心但最容易被忽略的。它决定了知识库是"活的"还是"死的"。

### 反哺事件类型

| 事件 | 触发条件 | 记录内容 |
|------|---------|---------|
| PRD 纠正 | 产品/开发发现 PRD 表述偏离业务意图 | 纠正记录 → 知识库更新 |
| 方案纠正 | 技术方案设计不合理被 review 打回 | 设计决策 → 模式库更新 |
| Bug 修复 | 线上/测试 bug 修复 | 根因分析 → 规则库更新 |
| 技能纠正 | skill 执行路径偏差被人工修正 | 技能模板更新 |

### 在 Kit 中的实现

```
AI 执行 skill ─→ 人发现错误 ─→ 人纠正 ─→ 记录到 corrections.log
    ↑                                              │
    │                   下次同类问题不会犯了         │
    └──────────────────────────────────────────────┘
```

实现模式可用 `agents/skills/resolve-feedback/`：
1. Agent 执行后有 trace 日志
2. 用户反馈正确/错误
3. 错误事件被记录为结构化数据（什么上下文、什么输出、正确是什么）
4. 需求上线后通过 `retro` skill 沉淀到知识库

### 反哺的 Kit 落地模式

```yaml
# workflow-history 中的反哺记录格式（参考）
correction:
  timestamp: "2026-06-20T14:30:00+08:00"
  trigger_skill: "dev-workflow"
  context: "合同管理 - 录入合同"
  incorrect_output: "创建了 draft 状态的合同但未调用审批"
  correct_action: "创建合同后需自动发起审批流（合同金额 > 0 时触发）"
  rule_update:
    - type: "knowledge"
      target: "agents/skills/contract-management/references/rules.md"
      change: "新增【服务端强制】合同创建后自动触发审批流"
  status: "applied"
```

## 反哺飞轮效应

知识库刚推出时没人爱用——扫得不准、规则缺失、AI 跑偏。但在强制使用的过程中：

1. 每用一次 → 纠一批错 → 补一点知识 → 磨一回 skill
2. 用得越多的 skill → 知识越厚 → AI 效果越好 → 大家自发用起来

**不是先做到完美再推广，而是在用的过程中一点点变好。**

---

## 与 system distillation 的关系

```
蒸馏方法论（S0-S6）──→ 知识库初始构建
                            │
                    扫描循环 → 持续更新事实
                    消费循环 → skill 执行时消费
                    反哺循环 → 经验沉淀提升知识质量
                            │
                            ▼
                    知识库从不完美到越来越好
```

蒸馏提供冷启动，三循环提供持续热更新。
