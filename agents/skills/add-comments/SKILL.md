# 添加注释

让下一个人能看懂代码，用清晰、简洁的上下文注释非显而易见的逻辑。

**专家角色：** 技术文档工程师 — 让下一个人能看懂代码。

**上游技能：** `dev-workflow` (Phase 4), `auto-complete` (output)
**下游技能：** `sync-progress`

---

## 何时使用

- "添加注释" / "完善注释" / "加注释"
- "add comments" / "improve comments"

## 何时不使用

- 代码已经自解释
- 添加注释只是重复显而易见的代码
- 用户只是想快速修复
- 时间紧迫

## 常见陷阱（先读这个！）

| 失败 | 预防 |
|------|------|
| 注释显而易见的代码 | **分层检查**：如果L1/L2已说明用途，代码逻辑清晰，不要添加L3逐行注释 |
| 注释与代码不匹配 | 添加后验证准确性 |
| 注释过于冗长 | 最多一句话 |
| **有了外层注释仍添加内层逐行注释** | **代码自解释时，L3注释应最小化** |
| 缺少业务上下文 | 关注WHY，不是WHAT |
| 复杂验证/重试逻辑只写表面描述 | 对失败阶段、验证边界、取舍原因必须写WHY |

---

## 红旗警告（停止）

| 想法 | 为什么错 |
|------|----------|
| "注释越多越好" | 注释增加噪音，不是价值 |
| "每行都要注释" | **分层原则：L1/L2充分时，L3应最小化** |
| "解释代码做了什么" | 代码展示WHAT，注释展示WHY |
| **"有了方法注释，每行也要注释"** | **冗余：方法注释已说明用途，代码自解释时不需要逐行注释** |

## 什么需要注释

| 类别 | 注释重点 |
|------|----------|
| 边界条件 | 为什么是这个边界？ |
| 业务规则 | 业务上下文 |
| 错误处理 | 错误策略 |
| 复杂条件 | 简化含义 |
| 非显而易见代码 | 为什么用这个方法？ |
| 验证/复验逻辑 | 为什么只验这一阶段？为什么不是全量？风险是什么？ |

## 注释分层

注释应该遵循**分层原则**，外层注释充分时，内层注释应尽量少。

| 层级 | 位置 | 内容 | 示例 |
|------|------|------|------|
| **L1 类级** | 类声明前 | 需求对应、核心职责、整体流程 | `/** 用户服务：处理用户注册、登录等 */` |
| **L2 方法级** | 方法声明前 | 用途、参数、返回值、设计说明 | `/** 验证用户名格式：长度4-20位，含字母数字 */` |
| **L3 行内** | 代码行前 | 仅解释非显而易见的WHY | `// Why: 防止用户名枚举攻击` |

**关键原则：**

- ✅ **L1/L2 充分 → L3 最小化**：类/方法注释已说明用途，代码逻辑清晰时不再添加行内注释
- ✅ **代码自解释 > 注释**：`userService.save(user)` 已说明做什么，不需要 `// 保存用户`
- ✅ **只注释WHY，不注释WHAT**：代码展示WHAT，注释解释WHY
- ❌ **避免注释冗余**：不要重复类/方法已说明的内容

## 原则

1. **简洁：** 默认最多一句话；安全/验证关键逻辑可放宽到两句
2. **准确：** WHAT 和 WHY，不是 HOW
3. **有用：** 解释非显而易见的东西
4. **适度：** 不是每行都需要注释
5. **分层优先：** L1/L2充分时，L3应最小化
6. **验证关键逻辑写 WHY：** 涉及失败阶段、局部验证、重试/回滚时，必须说明背后的约束或风险

---

## 术语约定（本技能）

- **Hard Gate（硬门槛）**：`关键 WHY 硬规则` 未满足时，不得把注释视为合格
- **PARTIAL**：本技能不使用 `PARTIAL` 表示注释完成度
- **完成标准（DoD）**：统一表示“需要注释的 WHY 已补齐且与代码一致”
- **验证范围**：在本技能中仅表示注释解释到哪一层验证边界，不表示测试执行状态

## 注释检查清单

在添加注释前，先问自己：

1. **L1类级注释是否已充分？**
   - [ ] 是否说明了核心职责？
   - [ ] 是否说明了与需求/设计文档的对应关系（如有）？

2. **L2方法级注释是否已充分？**
   - [ ] 是否说明了用途和设计决策？
   - [ ] 参数和返回值是否清晰？

3. **L3行内注释是否必要？**
   - [ ] 代码是否自解释？（`service.save(user)` 不需要 `// 保存用户`）
   - [ ] 是否解释了WHY而非WHAT？
   - [ ] 是否与L1/L2注释冗余？

**原则**：如果L1/L2已说明，且代码自解释，则不需要L3。

## 工作流程

```
[1] 获取范围（git diff / 文件 / 选择）
    ↓
[2] 检查现有注释 → 准确？跳过 : 更新
    ↓
[3] 识别缺失的注释
    ↓
[4] 用语言格式添加/更新
    ↓
[5] 输出摘要
```

**关键 WHY 硬规则：**

- 涉及重试策略、失败阶段复验、局部验证、回滚/降级逻辑时，禁止只写“在做什么”
- 注释至少要说明以下之一：
  - 为什么只覆盖这个阶段/范围
  - 为什么不能直接做全量或其它阶段
  - 这样做在防什么历史问题或系统风险
