# 自动完成模式

端到端交付，不中断，但知道何时升级。

**专家角色：** 高级工程师 — 端到端交付不中断，但知道何时升级。

**上游技能：** `req-alignment-check`, `pre-flight-check`
**下游技能：** `gen-tests`, `add-comments`, `deep-review`, `compound-learning`

---

## 何时使用

- "请你独自完成" / "全程自动" / "中间不要问我"
- "complete this autonomously" / "don't ask me questions"

## 何时不使用

- 任务需要创意决策 | 用户有空且想参与 | 高风险生产变更

---

## ⚠️ Mandatory Workflow

**When triggered, MUST follow these steps IN ORDER. Each step produces a deliverable.**

```
[0]  预检查 + 解析子任务 → 输出：任务清单
[0.5] 环境检测 → 输出：环境状态（缺失则暂停 #6）
[1]  隐藏需求扫描 → 输出：需求清单（缺口则暂停 #3）
[2]  架构检查 → 输出：一致性确认（冲突或上游设计升级则暂停 #1）
[3]  执行 + 修复循环 → 输出：实现代码（3次失败暂停 #4）
[4]  测试（gen-tests）→ 输出：测试报告（必须写明验证范围与原失败阶段复验）
[5]  提交 + Handoff Summary → 输出：交付摘要
```

**必须：** 执行此技能前使用 `pre-flight-check` 技能。

若项目存在 `.memory/build-test-profile.yaml`，后续测试阶段必须继承该 profile，不得在本技能中临时手拼另一套 Maven 验证命令。

## Stop Reason 枚举

| stop_reason | 含义 | 行动 |
|-------------|------|------|
| `completed` | Phase 正常完成 | 进入下一步 |
| `max_turns_reached` | 达到步数上限 | 暂停 #4 |
| `retry_limit` | 修复3次失败 | 暂停 #4 + compound-learning |
| `gate_failed` | Iron Law 触发 | 暂停对应 # |
| `user_cancelled` | 用户中断 | 保存进度，退出 |
| `env_error` | 环境异常 | 暂停 #6 |
| `DONE_WITH_CONCERNS` | 完成但有遗留 | 推荐下一步 + 列出遗留 |
| `NEEDS_CONTEXT` | 需要更多信息 | 暂停等待用户输入 |
| `BLOCKED` | 无法继续 | 三级升级：补充上下文 → 换策略 → 人工 |
| `escalate_upstream` | 发现上游设计问题 | 回溯到上游技能 |
| `replan_required` | 实现中发现设计输入升级 | 回退 `deep-plan`，更新设计后再继续 |

**输出格式：** `[STOP: completed]` 或 `[STOP: retry_limit]` 等。

---

## 统一状态码

完成后输出标准化状态（供 `workflow-router` 消费）：

| 状态码 | 含义 | 后续路由 |
|--------|------|---------|
| `DONE` | 全部子任务完成 | → gen-tests |
| `PARTIAL` | 仅部分验证完成，不得进入交付链路 | → 回到验证/审查 |
| `DONE_WITH_CONCERNS` | 完成但有遗留项 | → deep-review（优先检查遗留）|
| `NEEDS_CONTEXT` | 缺信息无法继续 | 等用户输入 |
| `BLOCKED` | 遇到无法解决的阻塞 | 三级升级：补充→换策略→人工 |
| `ESCALATE_UPSTREAM` | 发现上游设计问题 | → deep-plan（带错误上下文）|

**升级协议：** BLOCKED 时：1) 补充上下文重试 → 2) 换策略/拆小任务 → 3) 升级给用户。每级1次尝试。

## 上下文管理（Auto-Compaction）

当上下文超过阈值时主动 compact：
- **触发：** 模型重复 / 遗忘 / 工具截断
- **保留：** 任务清单 + 已完成项 + 关键决策 + stop_reason
- **丢弃：** 中间调试过程 / 失败尝试日志

## 常见陷阱

| 失败 | 预防 |
|------|------|
| "用户说不要问" → 错误假设 | 不覆盖 Iron Law |
| 选第一个选项 → 错过更好方案 | 在暂停点展示选项 |
| 跳过架构检查 → 不一致 | 总是先检查模式 |
| 假设隐藏需求 → bug | 不在规格中就问 |
| 无限修复循环 | 修复限3次，然后升级 |
| 提交太晚 → 丢失进度 | 每个子任务通过后提交 |
| 过滤测试通过就自动收尾 | 必须复验原失败阶段，局部通过不等于可交付 |
| 自动执行时把上游设计问题吞掉 | 遇到外部契约 / DTO / 配置迁移 / 扩展性新约束时强制升级 |

---

## Iron Law（铁律）

**安全规则：**
暂停当遇到以下情况 — 架构不一致、需求模糊、隐藏需求未覆盖、重大风险、修复循环超过3次、环境异常。
没有调查就没有修复 — 追踪数据流 → 识别根因 → 提出假设 → 修复上限 3 次。
若本轮或上一轮出现 `resolve / compile / testCompile / package` 失败，进入 Step [5] 前必须复跑并通过同一失败阶段或项目约定的完整验证命令。
若实现阶段出现外部契约未冻结、DTO 边界变化、配置迁移策略变化、未来扩展边界变化，必须返回 `ESCALATE_UPSTREAM` 或 `[STOP: replan_required]`，而不是继续自动补丁式推进。

**破坏性操作前暂停：** `rm -rf`, `git push --force`, `DROP TABLE`, 不可逆迁移

---

## 术语约定（本技能）

- **Hard Gate（硬门槛）**：暂停点、Iron Law 与 `Step [4] Hard Gate` 都属于 Hard Gate，未通过不得收尾
- **PARTIAL**：只表示部分验证，不能表示“已经基本完成”
- **完成标准（DoD）**：统一表示“代码实现、验证证据、交付摘要都已满足”的条件
- **验证范围**：统一使用 `文件 / 模块 / workspace / 全仓库`

---

## 红旗（停止）vs 绿旗（继续）

| 停止 | 绿旗 |
|------|------|
| "我选一个"（模糊） | 遵循模式（一致性）|
| "更简单的解释"（假设）| 文档化选择（可纠正）|
| "风险很小"（任何风险）| 只是命名（次要）|

## 常见借口

| 借口 | 反击 | 行动 |
|------|------|------|
| "用户说不要问" | Iron Law 优先于用户偏好 | 暂停 #1-#6 |
| "这是合理的假设" | 未验证 = bug 源头 | 验证或暂停 |
| "差不多就行" | 差不多 = 技术债 | 跑测试贴证据 |
| "修了3次还不行就跳过" | 跳过 = 遗留缺陷 | PAUSE #4 + compound-learning |
| "这个风险很小" | 小风险累积成事故 | 列入 Handoff Summary |
| "用户不在，先推了" | 未审查代码不上远程 | 保留本地，等确认 |
| "环境问题等会再修" | 环境阻塞 = 无法验证 | PAUSE #6 |
| "我先随手跑个 mvn test" | 偏离项目 profile 可能制造假阴性/假阳性 | 先读 `.memory/build-test-profile.yaml` |
| "这个模式大家都这样写" | 一致 ≠ 正确 | 检查是否有更好方案 |
| "先实现再说，测试后面补" | 后面 = 永远不 | Phase 4 必须测试 |
| "只改了一行不用审查" | 一行也能引入 P0 | 完整 review 不跳步 |

**Spirit-over-Letter 声明：** Iron Law 是安全网不是形式主义。当字面规则与安全精神冲突时，遵守精神 — 宁可多暂停一次，不可带病交付。

## 能动性等级

| 行为 | 被动 ❌ | 主动 ✅ |

## 6个暂停点

| # | 触发 | 输出 |
|---|------|------|
| 1 | 架构不一致 / 上游设计升级 | 模式 vs 计划 → 用哪个？ |
| 2 | 需求模糊 | 原始 + 解释 → 选哪个？ |
| 3 | 隐藏需求缺口 | 类别 + 选项 → 如何处理？ |
| 4 | 修复上限（3次）| 尝试 + 诊断 → 需要帮助 |
| 5 | 子任务冲突 | 冲突变更 → 保留哪个？ |
| 6 | 环境异常 | 依赖缺失/权限 → 需用户处理 |

## 上游设计升级触发器

自动执行中出现以下任一信号，必须停止当前实现并升级到 `deep-plan`：

- 外部接口字段开始讨论“顶层还是 detail”、“通用壳还是来源专属 DTO”
- 用户或代码现状暴露出新兼容性要求、新调用方、未来新增来源复用要求
- 新增 Apollo key、namespace、缓存刷新、fallback、旧 key 兼容/清理策略突然成为关键问题
- 开发中发现“为什么没沿用旧实现”的问题其实是设计取舍，而不是编码遗漏

**执行规则：**

- 自动模式不等于忽略这些问题
- 命中上述信号后，输出 `ESCALATE_UPSTREAM` 或 `[STOP: replan_required]`
- 附带最小上下文：新增约束、影响范围、当前实现为什么不能继续硬冲

**推荐动作：**

- 外部 DTO / 契约问题 → 回到 `deep-plan`，执行“外部集成 DTO 七问”
- 配置迁移问题 → 回到 `deep-plan`，加载 `config-migration-checklist.md`

## 环境异常自动识别

| 异常 | 检测信号 | 自动处理 | 暂停时机 |
|------|----------|----------|----------|
| 依赖缺失 | `Cannot find module` | 尝试安装 | 安装失败 |
| 权限不足 | `EACCES` / `Permission denied` | 无 | 立即 |
| 端口占用 | `EADDRINUSE` | 尝试 kill | 无法安全 kill |
| 环境变量 | `undefined` 关键配置 | 检查 .env | 需用户密钥 |

---

## 优雅退出

暂停时输出：尝试方案 + 卡点 + 用户选项。

---

## 隐藏需求扫描

**总是检查：** 错误处理、边界情况、输入验证
**按上下文：** 异步/API → 幂等性；共享状态 → 并发；UI → 状态管理/无障碍

---

## 并行策略 & 修复循环

- ≥2个独立子任务 → Agent 工具启动子代理（最多3并发）
- 每个子代理：实现 → 测试 → 修复循环 → 隔离提交
- 修复失败3次 → PAUSE #4

**决策规则：** 最简单 → 最安全 → 一致 → 明确 → 文档化

---

## Handoff Summary

核心要素：建了什么、怎么验证、关键决策、已知缺漏。

**Step [4] Hard Gate：**

- 测试报告必须写明：执行命令、验证范围（文件 / 模块 / workspace / 全仓库）
- 若之前失败的是 `compile` / `testCompile` / `package`，必须写明该阶段在本轮已重新通过
- 只跑过滤测试 = 最多算 `PARTIAL`
- `PARTIAL` 不得进入 Step [5] 提交 + Handoff
- 若本轮中途曾命中上游设计升级但未回退规划，不得进入 Step [5]
