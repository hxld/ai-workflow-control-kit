# Dialogue Learning

从对话中学习，识别知识薄弱点和解决模式，让AI越用越懂你。

**专家角色：** 学习伙伴 — 从对话中识别你的认知缺口和解决模式。

**上游技能：** 所有对话（被动监听）
**下游技能：** `req-alignment-check`, `auto-complete`, `restore-context`, `skill-evolution`

---

## 何时使用

- 用户问"什么是..."/"怎么理解..."/"...是什么意思"（知识问答）
- 用户说"记录这个模式"/"标记薄弱点"/"查看薄弱点"/"查看模式"（手动触发）
- 用户说"记录下来"/"记成经验"/"记录经验"/"总结成经验"/"沉淀一下"（通用沉淀触发）
- 对话结束后自动识别（自动模式）

## 何时不使用

- 简单的指令执行（"帮我写个函数"）
- 明确的开发任务（已由其他技能处理）

## Iron Law（铁律）

- 只记录"问什么"，不记录"谁问的"——隐私优先，不包含业务敏感信息。
- 3次=薄弱点，5次=建议转技能，严格执行。
- 对“过早宣称完成 / 局部验证冒充整体完成”的重复纠正，必须按流程模式记录，不得只当普通问答跳过。

---

## 术语约定（本技能）

- **Hard Gate（硬门槛）**：分类错误会污染后续提醒，因此“验证纪律专项规则”和阈值处理属于 Hard Gate
- **PARTIAL**：本技能不使用 `PARTIAL` 表示学习进度；若只捕捉到一次线索，只做计数，不写成部分完成状态
- **完成标准（DoD）**：统一表示“已正确分类、已更新计数、已判断是否触发下游提醒/转技能”
- **验证范围**：在本技能中主要表示验证纪律类模式涉及的校验边界，不表示测试执行状态

---

## 常见陷阱（先读这个！）

| 失败 | 预防 |
|------|------|
| 记录太频繁 → 噪音 | 只记录真正的知识问答/模式 |
| 分类不准 → 难检索 | 严格按技术栈分类 |
| 只记录不应用 → 没用 | 在其他技能中主动参考 |
| 阈值太低 → 误判 | 3次才标记，5次才建议 |
| 反复被纠正“局部通过=整体完成”却没沉淀 | 将其记录为流程/验证边界模式，不只当知识问答 |

---

## 红旗警告（停止）

| 想法 | 为什么错 |
|------|----------|
| "多记录点总没错" | 噪音会淹没有价值的信息 |
| "这个问题很重要" | 重要 ≠ 反复出现，不要强制记录 |
| "用户肯定想知道" | 不经确认不要假设用户需求 |

---

## 常见借口

| 借口 | 反击 | 行动 |
|------|------|------|
| "这个知识很基础" | 基础 = 可能反复问 | 记录 |
| "只问了一次" | 一次也可能重要 | 计数，不标记 |
| "模式太简单" | 简单 = 高复用 | 记录 |

## 工作流程

```
[1] 检测对话类型
    ├── 知识问答 → 提取问题 + 分类 → [2a]
    ├── 解决模式 → 提取模式 + 分类 → [2b]
    ├── 纠偏模式 → 用户多次纠正“过早完成声明/局部验证冒充整体” → [2b]
    ├── 手动触发 → 直接记录 → [2a/b]
    └── 普通对话 → 跳过
    ↓
[2a] 更新 knowledge-gaps.md（计数+1）
[2b] 更新 solution-patterns.md（计数+1）
    ↓
[3] 检查阈值 → 3次标记薄弱点 / 5次建议转技能
```

## 自动检测规则

### 知识问答

| 模式 | 示例 |
|------|------|
| `什么是X` / `X是什么意思` / `怎么理解X` | "什么是CAS" |
| `X和Y的区别` / `X的原理` | "HashMap和ConcurrentHashMap区别" |

### 解决模式

| 模式 | 示例 |
|------|------|
| 反复出现的问题（2次+）| 同类错误 |
| 固定的解决步骤 | "日志分析5步法" |
| 用户说"记住这个" | "记住这个排查方法" |
| 重复纠正代理边界错误 | "局部测试通过不等于整体修复完成" |

---

## 阈值处理

### 薄弱点（3次）

标记后自动：
1. 在 `req-alignment-check` Phase 1 主动提示
2. 在 `auto-complete` 隐藏需求扫描时参考
3. 在 `dev-workflow`、`sync-progress`、`resolve-feedback` 中作为验证纪律提醒

**提示示例：**
```
检测到你在【并发】领域有薄弱点：CAS原理（问过3次）
建议额外关注：是否需要考虑并发安全？是否涉及原子操作？
```

### 转技能（5次）

提示用户确认 → 调用 `skill-evolution`

## 与其他技能集成

| 技能 | 集成点 |
|------|--------|
| **req-alignment-check** | Phase 1 参考薄弱点 |
| **auto-complete** | 隐藏需求扫描参考薄弱点 |
| **restore-context** | 恢复时加载薄弱点摘要 |
| **skill-evolution** | 模式达阈值时调用 |
| **compound-learning** | 区分"错误"和"知识缺口"，并将技能问题写入 `skill-feedback.md` |
| **dev-workflow / sync-progress** | 若存在“验证纪律”高频模式，优先提醒失败阶段复验与验证范围 |
| **deep-review / quality-check / resolve-feedback** | 若存在“验证纪律”模式，优先检查是否把局部验证误当整体完成 |

## 验证纪律专项规则

若用户反复纠正以下模式：

- 局部测试通过 = 整体修复完成
- 未复跑原失败阶段就宣称完成
- 没说验证范围就给整体结论

则必须：

1. 归类为 `验证纪律`，不是普通知识问答
2. 写入 `solution-patterns.md`
3. 达到阈值后在相关下游技能中优先提示

## 手动触发

| 命令 | 动作 |
|------|------|
| `记录这个模式` | 提取当前对话为模式 |
| `标记薄弱点` | 强制标记为薄弱点 |
| `查看薄弱点` | 输出薄弱点列表 |
| `查看模式` | 输出高频模式列表 |
| `记录下来` / `记成经验` / `记录经验` / `总结成经验` / `沉淀一下` | 以通用表达触发当前对话的学习沉淀 |
