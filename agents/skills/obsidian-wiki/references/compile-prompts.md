# Compile Prompts — 按来源类型的提取模板

## 使用方式

ingest 时根据来源类型选择对应模板，执行两阶段编译：
1. **Per-source pass**：从单个来源提取结构化信息
2. **Cross-source pass**：将提取结果整合到现有 wiki 页面

---

## Per-Source Pass 模板

### technical（接口/技术文档）

**触发条件：** 文件名或内容含"接口"、"API"、"系统"、"架构"、"接口文档"

**提取目标：**
- 系统架构概述（组件、依赖关系）
- 接口定义（端点、请求/响应、数据结构）
- 数据流（输入→处理→输出）
- 技术决策（为什么这样设计、考虑了哪些替代方案）
- 约束条件（性能、安全、兼容性）
- 关键概念（需要创建 concept 页面的术语）
- 关键实体（需要创建 entity 页面的系统/组件/团队）

**输出格式：**
```
## 摘要
[2-3 句话概述]

## 系统架构
[描述涉及的系统和组件]

## 关键接口/功能
- [功能1]：[简述]
- [功能2]：[简述]

## 技术决策
- [决策1]：[原因]
- [决策2]：[原因]

## 提取的概念
- [[concept-1]]: 一句话定义
- [[concept-2]]: 一句话定义

## 提取的实体
- [[entity-1]]: 一句话描述
- [[entity-2]]: 一句话描述

## 关键要点
1. [要点1]
2. [要点2]
3. [要点3]
```

### meeting（会议记录）

**触发条件：** 文件名或内容含"会议"、"周报"、"日报"、"评审"、"同步"

**提取目标：**
- 会议背景和目的
- 关键决策（做了什么决定）
- 待办事项（谁负责、截止时间）
- 时间线（关键里程碑）
- 风险和阻塞
- 提及的实体（人名、系统名、项目名）

**输出格式：**
```
## 摘要
[1-2 句话]

## 关键决策
- [决策1]
- [决策2]

## 待办事项
- [ ] [任务1] — @负责人 — 截止 YYYY-MM-DD
- [ ] [任务2] — @负责人

## 风险/阻塞
- [风险1]

## 提取的实体
- [[entity-1]]: 角色/描述
```

### requirement（需求/PRD/设计文档）

**触发条件：** 文件名或内容含"需求"、"PRD"、"设计"、"方案"、"迭代"

**提取目标：**
- 需求背景和目标
- 核心需求点列表（优先级）
- 涉及的系统/模块
- 业务流程（输入→处理→输出）
- 状态（规划中/开发中/已完成）
- 风险点
- 验收标准

**输出格式：**
```
## 摘要
[2-3 句话]

## 需求背景
[为什么要做这个需求]

## 核心需求
- P0: [需求1]
- P1: [需求2]
- P2: [需求3]

## 涉及系统
- [[entity-system-1]]: [角色]
- [[entity-system-2]]: [角色]

## 业务流程
1. [步骤1]
2. [步骤2]

## 验收标准
- [ ] [标准1]
- [ ] [标准2]

## 提取的概念和实体
...
```

### paper（论文）

**触发条件：** .pdf + 文件名或内容含"论文"、"arxiv"、"研究"、"paper"

**提取目标：**
- 核心贡献（1-3 句话）
- 方法论（模型架构、训练策略）
- 数据集和实验设置
- 主要结果（关键指标）
- 局限性
- 后续工作方向

**输出格式：**
```
## 摘要
[2-3 句话]

## 核心贡献
1. [贡献1]
2. [贡献2]

## 方法论
[方法描述]

## 实验结果
- [指标1]: [数值]
- [指标2]: [数值]

## 局限性
- [局限1]

## 与其他来源的关系
- 补充了 [[concept-x]] 的 [方面]
- 挑战了 [[concept-y]] 的 [假设]

## 提取的概念和实体
...
```

### book（书籍/课程）

**触发条件：** 文件名或内容含"书"、"章"、"课程"、"教程"、"第X章"

**提取目标：**
- 核心论点/主题
- 章节结构和关键论证
- 重要概念解释
- 实践建议/可操作要点
- 与其他知识的关联

**输出格式：**
```
## 摘要
[2-3 句话]

## 核心论点
[作者的主要观点]

## 关键章节
- 第X章 [标题]: [核心论点]
- 第Y章 [标题]: [核心论点]

## 重要概念
- [[concept-1]]: 定义 + 为什么重要
- [[concept-2]]: 定义 + 为什么重要

## 实践建议
1. [建议1]
2. [建议2]

## 提取的概念和实体
...
```

### travel（旅行攻略）

**触发条件：** 文件名或内容含"旅行"、"攻略"、"行程"、"旅游"、"酒店"

**提取目标：**
- 目的地信息
- 行程安排（日期、地点、活动）
- 预算信息
- 交通方式
- 住宿推荐
- 经验总结

**输出格式：**
```
## 摘要
[1-2 句话]

## 目的地
[城市/国家] — [简介]

## 行程
| 日期 | 活动 | 地点 | 备注 |
|------|------|------|------|
| ... | ... | ... | ... |

## 预算
- 交通：[金额]
- 住宿：[金额]
- 餐饮：[金额]
- 门票：[金额]

## 经验总结
1. [经验1]
2. [经验2]

## 提取的实体
- [[entity-place-1]]: 描述
```

### article（通用文章）

**触发条件：** 不匹配以上任何类型

**提取目标：**
- 核心观点（1-3 个）
- 论证逻辑
- 关键数据/案例
- 与已有 wiki 内容的关联（补充/挑战/无关）

**输出格式：**
```
## 摘要
[2-3 句话]

## 核心观点
1. [观点1]
2. [观点2]

## 关键论据
- [论据1]
- [论据2]

## 与已有知识的关系
- 补充了 [[concept-x]]
- 挑战了 [[concept-y]]

## 提取的概念和实体
...
```

---

## Cross-Source Pass 规则

Per-source 提取完成后，将结果整合到现有 wiki：

1. **概念页**：如果 concept 已存在，追加新来源的视角（标注来源差异）；如果不存在，创建新页面
2. **实体页**：如果 entity 已存在，更新描述（合并信息）；如果不存在，创建新页面
3. **矛盾处理**：如果新来源与已有页面矛盾，在页面中标注"⚠️ 矛盾"并列出不同来源的观点
4. **跨领域链接**：如果新概念涉及其他领域，在目标领域创建 stub 页面（带 `stub: true` frontmatter）
