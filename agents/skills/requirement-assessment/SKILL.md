# 需求评估

评估业务需求合理性 — 基于数据与事实筛选高价值需求，不做传声筒，要做过滤器。

**专家角色：** 需求过滤器 — 基于数据与事实筛选高价值需求。

**下游技能：** `req-alignment-check`, `deep-plan`

---

## 何时使用

- 收到业务方需求文档请求评估
- 产品经理收到需求需要判断是否值得深入
- 需求评审会议前的预筛选
- "评估需求" / "需求评审" / "需求过滤"

## 何时不使用

- 已通过评审的需求 → 用 `req-alignment-check`
- 紧急 bug 修复 → 直接进入开发
- 技术债务清理 → 用 `quality-check` / `deep-review` 先评估风险和收益
- 已有明确 PRD → 用 `req-alignment-check` → `deep-plan`

## 常见陷阱

| 失败 | 预防 |
|------|------|
| 接受软证据为硬证据 | 必须区分证据级别 |
| 驳回后无改进路径 | 每次驳回必须给出具体建议 |
| 主观判断影响评分 | 严格按评分细则打分 |
| 忽略战略对齐 | 必须查阅当前 OKR |
| 评估耗时过长 | 单个需求 ≤ 15 分钟 |
| 已有人工流程就误判为低价值 | 先看自动化是否能替代高频人工协同与重复维护 |
| 价值高就放松完成标准 | 高价值不等于可直接通过，验收和上线门槛必须明确 |

---

## 红旗信号（一票否决）

| 信号 | 常见辩解 | 反驳 |
|------|----------|------|
| 无法提供具体问题案例 | "大家都知道这个问题" | 无案例 = 伪需求 |
| 影响人数 < 5 人且非核心业务 | "这几个是大客户" | 大客户应有专项流程 |
| "别人都有所以我们也要有" | "竞品有这功能" | 你的用户需要吗？ |
| 收益完全无法量化 | "提升用户体验" | 无法量化 = 无法验证 |
| 与战略方向冲突 | "先做了再说" | 资源有限，聚焦重点 |

---

## 常见借口

| 借口 | 反击 | 行动 |
|------|------|------|
| "这个需求很急" | 紧急 ≠ 重要 | 评估优先级而非速度 |
| "业务方已经确认了" | 业务确认 ≠ 需求合理 | 独立评估，不盲从 |
| "先做 MVP 看看" | MVP 也需要验证必要性 | 先验证问题真实性 |
| "这个很简单" | 简单 ≠ 有价值 | 评估收益而非难度 |
| "领导说的" | 领导也有盲点 | 用数据说话 |

---

## Iron Law（铁律）

- 无案例不评分，无数据不通过，无收益不开发。
- 评估前必须确认：需求文档可读、至少包含问题描述、未评审过。无法满足则返回追问清单。
- 若缺少可验收、可验证、可上线的完成定义，不得因业务价值高而直接判通过。

---

## 证据分级

| 级别 | 可信度 | 示例 |
|------|:------:|------|
| **硬证据** | 高 | 埋点数据、财务报表、工单记录、命令输出 |
| **软证据** | 中 | 用户访谈、业务方声明、竞品分析 |
| **无证据** | 低 | "我觉得"、"应该"、"可能" |

**评分规则：硬证据满分，软证据 60%，无证据 0 分**

---

## 评估框架（100 分）

| 维度 | 分值 | 核心检查点 |
|------|:----:|------------|
| 必要性 | 30 | 问题真实性(15) + 影响范围(10) + 当前应对(5) |
| 价值性 | 40 | 收益可量化(20) + 投入产出比(10) + 战略对齐(10) |
| 可行性 | 30 | 技术可行性(10) + 资源匹配(10) + 优先级(10) |

> 详细打分细则见 `references/scoring-rubric.md`

## 决策规则

| 得分 | 红旗 | 结论 | 下一步 |
|:----:|:----:|------|--------|
| ≥ 80 | 无 | ✅ 通过 | 提交产品经理评审 |
| 60-79 | 无 | ⚠️ 待讨论 | 安排产品 + 业务沟通会 |
| < 60 | 无 | ❌ 驳回 | 补充材料后重新提交 |
| 任意 | 有 | ❌ 驳回 | 说明红旗原因，给改进路径 |

## 交付质量 Hard Gate（硬门槛）

以下内容缺失时，结论最高只能是 `⚠️ 待讨论`，不得直接 `✅ 通过`：

- 验收标准：如何判断做完
- 验证范围：要跑哪些测试/检查/链路
- 失败阶段复验：若历史失败过，是否要求复跑对应阶段
- 上线门槛：发布/放量/回滚条件

**评分惩罚：**

- 缺 1 项 → 总分最多 79
- 缺 2 项 → 总分最多 69
- 缺 3 项及以上 → 总分最多 59

也就是说：即使业务价值很高，交付质量定义严重缺失时，分数本身也必须显著下降。

---

## 术语约定（本技能）

- **Hard Gate（硬门槛）**：本技能中的入口条件与交付质量约束都属于 Hard Gate，未满足不得判定 `✅ 通过`
- **完成标准（DoD）**：统一表示“如何判断做完 / 可验收 / 可上线”
- **验证范围**：统一表示要跑哪些测试、检查与链路覆盖

---

## 系统集成/流程自动化类需求快速判断

当需求属于“已有人工处理、已有部分链路、现在补自动同步/自动打通”时，额外检查：

- 当前人工量是否持续发生、是否跨团队协同
- 是否已有后半段能力可复用，只缺前半段自动打通
- 收益是否体现在时效、出错率、客户体验和人工节省
- 是否因为系统集成缺口导致长期人工兜底

这类需求通常不是“新功能从 0 到 1”，而是“把已存在的人工断点系统化”，不要低估其业务价值。

## 工作流程

```
[1] 确认入口 Hard Gate 条件
    ↓
[2] 解析需求文档
    ↓
[3] 逐项评估，标注证据级别
    ↓
[4] 检查红旗信号（一票否决）+ 交付质量 Hard Gate
    ↓
[5] 计算总分 → 生成评估报告
    ↓
[6] 给出改进建议或下一步
```

## 追问清单（信息不足时）

| 维度 | 追问问题 |
|------|----------|
| 必要性 | 能否提供 3-5 个具体问题案例？影响多少人？现在怎么解决的？ |
| 价值性 | 预期收益？能否量化？预估成本？ROI？符合哪个 OKR？ |
| 可行性 | 技术难点？外部依赖？期望上线时间？排期允许吗？ |

---

## 何时暂停

暂停并询问用户：

1. 需求文档信息严重不足，追问后仍无法评估
2. 检测到红旗信号但业务方争议很大
3. 评分在 55-65 分边界，结论不明确
4. 需求与多个 OKR 相关，战略对齐不清晰
