# PRD 起草辅助与纠错反哺机制

> 来源：刘宇帅《AI 工作流跑不起来，问题可能出在PRD》
> 适配：ai-workflow-control-kit `requirement-assessment` skill 的可选扩展模式

## 概述

PRD 起草辅助是 `requirement-assessment` skill 的可选扩展模式。当用户说"帮我写个 PRD"或"起草需求文档"时，技能从"评估者"切换到"起草辅助者"角色。

**核心原则**：
- PRD 是一个一次性产物——写完、评审完、开发完，它的使命基本结束
- 代码是唯一的事实层——不要试图维护一份"永远最新"的 PRD
- 起草辅助的目标是**降低起草门槛、提高源头质量**，而非创造永续文档

---

## 模式切换

`requirement-assessment` 的默认模式是**评估模式**（分析已有需求）。
当触发以下信号时切换到**起草辅助模式**：

| 触发信号 | 示例 |
|---------|------|
| 用户直接要求 | "帮我起草一个 PRD" / "写个需求文档" |
| 有 BRD 但无 PRD | "有一份 BRD（商业需求文档），帮我转成 PRD" |
| 有口述需求 | "我想做个 XX 功能，帮我整理成 PRD" |
| PRD 质量体检不通过后 | "体检没通过，帮我补充缺失部分" |

---

## 起草辅助流程

```
用户输入（一句话 / BRD / 口述）
    ↓
[1] 理解业务意图
    ↓
[2] 套用 PRD 模板 → 生成草稿
    ↓
    ├─ 模式 A：从零起草（只有一句话）
    │     → 用知识库 + 代码理解补全上下文（如适用）
    │     → 产出带占位符的草稿
    │
    ├─ 模式 B：BRD → PRD 转化
    │     → 将 BRD 展开为标准 PRD 格式
    │     → 补充缺失段落
    │
    └─ 模式 C：口述 → PRD 整理
          → 将零散口述整理为结构化 PRD
          → 标记模糊点待确认
    ↓
[3] PRD 三层质量自查（调用 prd-quality-check 流程）
    ↓
[4] 输出草稿 + 质量体检报告
    ↓
[5] 用户确认/修改 → 最终版
```

---

## 起草辅助模式的关键决策点

### 何时需要"代码理解"

仅在以下条件都满足时，PRD 起草辅助才读取代码：
1. 用户显式授权（"可以看代码"或"用知识库"）
2. PRD 涉及已有系统的**功能变更或增强**
3. 有明确的业务上下文可以定位到代码

**不做的事情**：
- ❌ 不为纯粹的"新需求从零到一"读取代码（没有已有逻辑可参考）
- ❌ 不自动 clone 代码库（需要用户确认路径）
- ❌ 不将代码细节直接复制到 PRD（PRD 是业务语言，不是技术文档）

### PRD 的真实性边界

| PRD 中的内容 | 来源 | 可信度 |
|-------------|------|--------|
| 业务目标、背景 | 用户提供或 BRD | 与用户输入一致 |
| 系统现状描述 | 知识库 + 代码扫描 | 按知识库三级可信度 |
| 涉及系统清单 | 用户确认 | 用户确认即锁定 |
| 验收标准 | 起草辅助生成 + 用户调整 | 需用户最终确认 |

## 纠错反哺机制

### 适用场景

当 PRD 在评审或开发过程中被发现有问题时（写错/遗漏/口径不一致），通过反哺机制记录纠正内容，防止下次同类需求再犯。

### 反哺流程

```
开发/评审中发现 PRD 问题
    ↓
记录纠正事件：
  - 什么 PRD
  - 什么错误
  - 正确的应该是什么
  - 根因分析（模板不足 / 领域陌生 / 口径未对齐）
    ↓
需求上线后 → 反哺回知识库
    ↓
下次起草同类 PRD 时 → 主动引用历史纠正
```

### 纠正事件记录格式

```yaml
# workflow-history 中的 PRD 纠正事件
- prd_correction:
    timestamp: "2026-06-20"
    prd_title: "<PRD 标题>"
    prd_source: "<BRD/口述/业务方>"
    error_type: "missing_section / inaccurate_description / domain_ambiguity"
    incorrect: "原来的描述"
    correct: "纠正后的描述"
    root_cause: "模板校验_未覆盖 / 领域知识_缺失 / 口径未对齐"
    rule_update:
      - type: "template_check"
        change: "在规范层体检中新增检查点"
      - type: "knowledge_domain"
        target: "references/<domain>/prd-standards.md"
        change: "记录该领域的口径约定"
    status: "applied / pending"
```

### 反哺的三种级别

| 级别 | 触发条件 | 反哺动作 | 受益范围 |
|------|---------|---------|---------|
| L1：即时 | 单个 PRD 的单个错误 | 当前 PRD 修改 + 下次提醒 | 该 PRD 的下次迭代 |
| L2：模式 | 同一类错误出现 2+ 次 | 更新 PRD 质量体检检查项 | 所有后续 PRD |
| L3：领域 | 领域口径反复冲突 | 更新该领域的 references/ | 该领域所有 skill |

## 使用边界

| 要做 | 不要做 |
|------|--------|
| 降低 PRD 起草门槛 | 替代产品经理的业务判断 |
| 套用模板补全结构 | 编造用户没说过的业务逻辑 |
| 用已有知识库补全上下文 | 擅自访问无授权的代码 |
| 记录纠正事件 | 未经确认自动修改已完成的 PRD |
| 标记模糊点待确认 | 自作主张填上模糊点 |
