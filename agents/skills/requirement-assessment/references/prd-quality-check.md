# PRD 质量检测：三层体检

> 来源：刘宇帅《AI 工作流跑不起来，问题可能出在PRD》
> 适配：ai-workflow-control-kit `requirement-assessment` skill

## 概述

PRD 质量是 AI 工作流质量的源头。"垃圾进，垃圾出"——源头的 PRD 含糊，后面 AI 再能干也只是替你把含糊'发扬光大'。

本指南定义了 PRD 三层质量体检框架。**注意：本检测不碰业务逻辑，只做"规范层"的体检。**

## 三层体检模型

```
                ┌──────────────────┐
                │  ① 规范层        │  ← 合不合模板（快速排查）
                │  (模板完整性)     │
                └───────┬──────────┘
                        ▼
                ┌──────────────────┐
                │  ② 完备层        │  ← 细节够不够全（关键）
                │  (诉求清晰度)     │
                └───────┬──────────┘
                        ▼
                ┌──────────────────┐
                │  ③ 自洽层        │  ← 前后打不打架（深入）
                │  (内部一致性)     │
                └──────────────────┘
```

**原则**：三层依次递进。规范层不通过则退回补模板，不必继续深检。

---

## 第一层：规范层（模板完整性）

检查 PRD 是否符合标准模板，必要段落是否齐全。

### 检查项

| 检查点 | 说明 | 通过标准 |
|--------|------|---------|
| 项目背景 | 为什么要做这个需求 | 有明确的问题/机会描述 |
| 业务目标 | 要达到什么效果 | 有可衡量的目标（如"将 X 从 Y% 降到 Z%"） |
| 涉及部门 | 哪些团队参与 | 至少列出主责和协作方 |
| 影响系统 | 影响哪些服务/模块 | 点出系统名称，不要求精确 |
| 验收标准 | 怎么判断做完 | 有可验证的条件（如"用户可在列表页看到 X"） |
| 时间期望 | 期望什么时间上线 | 有目标日期或版本号 |

### 计分规则

| 结果 | 解释 | 分值 |
|------|------|:----:|
| ✅ 全部通过 | 模板完整 | +3 |
| ⚠️ 少量缺失 | 缺 1-2 项 | +1 |
| ❌ 严重缺失 | 缺 3 项及以上 | -3 |
| ❌ 无法评估 | 没有 PRD、只有口述或聊天记录 | 不适用本体检，退回要求书面化 |

### 输出

```
规范层：[通过 / 部分缺失 / 不通过]
缺失项：<逐项列出，每项附改进建议>
```

---

## 第二层：完备层（诉求清晰度）

检查细节是否到位——有没有把诉求真正讲清楚，而不是靠"你懂的"。

### 检查项

| 模糊信号 | 示例 | 应该长这样 |
|---------|------|-----------|
| 模糊词汇 | "优化体验"、"提升效率"、"更好的方式" | 具体的交互变更、预期效果 |
| 依赖"意会" | "和 XX 功能类似"、"你懂的" | 明确的业务描述，不依赖参照物 |
| 半截描述 | "需要个弹窗……"（没说完） | 完整的触发条件、内容、操作 |
| 替代原文 | "就是那个……"、"之前讨论过的" | 将"意会"内容写进文档 |
| 数据无来源 | "很多用户反馈" | 具体数据：多少人、什么渠道、时间 |

### 深入检查

当 PRD 涉及业务逻辑变更时，额外检查：

| 检查点 | 模糊 | 清晰 |
|--------|------|------|
| 规则定义 | "按规则处理" | "金额 > 1000 走二级审批，否则自动通过" |
| 边界行为 | 未提及 | 明确 null/空/极限情况处理 |
| 错误处理 | 未提及 | 明确失败时的用户可见提示和系统行为 |
| 异常流程 | 只写了主流程 | 至少列出 2-3 个异常分支 |

### 计分规则

| 结果 | 解释 | 分值 |
|------|------|:----:|
| ✅ 清晰 | 诉求自描述，无需要追问的模糊点 | +4 |
| ⚠️ 少量模糊 | 1-2 处需要追问 | +0 |
| ❌ 大量模糊 | 3 处以上模糊或关键逻辑缺失 | -4（模糊项数 ×-1，上限 -4） |

### 输出

```
完备层：[通过 / 待完善 / 不通过]
模糊项：<逐项列出追问问题>
```

---

## 第三层：自洽层（内部一致性）

检查 PRD 自身是否前后矛盾。

### 常见矛盾模式

| 矛盾类型 | 示例 |
|---------|------|
| 前后口径不一致 | 前面说"只影响 A 系统"，后面需求详情涉及 B、C 系统 |
| 数据冲突 | 前面说"日活 10 万"，后面说"影响 100 人" |
| 目标与方法矛盾 | 目标是"降低成本"，方案却要增加服务器 |
| 时间线矛盾 | 前面说"两周上线"，后面拆的任务量需要两个月 |
| 规则冲突 | 一条规则说"金额 > 1000 需审批"，另一条说"所有订单自动通过" |

### 计分规则

| 结果 | 解释 | 分值 |
|------|------|:----:|
| ✅ 一致 | 无矛盾 | +3 |
| ⚠️ 轻微矛盾 | 1 处有矛盾但不影响核心逻辑 | +0 |
| ❌ 严重矛盾 | 2 处以上或核心逻辑矛盾 | -3（每处 -1.5，上限 -3） |

### 输出

```
自洽层：[通过 / 有轻微矛盾 / 有严重矛盾]
矛盾项：<逐项列出矛盾点及建议统一口径>
```

## 完整体检报告模板

```markdown
## PRD 质量体检报告

**PRD 标题：** <标题>
**体检时间：** <时间>

### 规范层：[通过 / 部分缺失 / 不通过]
- [ ] 背景：<通过/缺失>
- [ ] 目标：<通过/缺失>
- [ ] 涉及部门：<通过/缺失>
- [ ] 影响系统：<通过/缺失>
- [ ] 验收标准：<通过/缺失>
- [ ] 时间期望：<通过/缺失>
- 缺失项改进建议：<...>

### 完备层：[通过 / 待完善 / 不通过]
- 模糊项清单：
  1. <模糊位置> → <追问>
  2. <模糊位置> → <追问>
- 边界/异常覆盖：[完整 / 部分缺失 / 严重缺失]

### 自洽层：[通过 / 有轻微矛盾 / 有严重矛盾]
- 矛盾项清单：
  1. <矛盾位置 A> vs <矛盾位置 B> → <建议统一方向>
  2. ...

### 总体评价
- **PRD 健康度：** [优秀 / 良好 / 需返工 / 不可用]
- **建议：** <通过评估 / 修改后重新体检 / 退回重写>
```

## 与知识库反哺循环的联动

PRD 三层体检中发现的系统性问题应通过反哺循环沉淀：

1. **模式性问题**（如"多个 PRD 都缺验收标准"）→ 更新 `agents/reference/knowledge-base-design/` 中的 checklists
2. **领域特定模糊**（如"某个业务域常有口径冲突"）→ 记录到该领域的 references/ 中
3. **改进建议** → 下次同类 PRD 评估时主动引用历史纠正记录
