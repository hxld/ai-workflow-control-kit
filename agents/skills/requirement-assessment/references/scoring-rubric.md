# 详细打分细则

## 一、必要性验证（30 分）

### 1. 问题真实性（15 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 15 | 提供 5+ 个具体案例 | 有时间/人物/场景/问题表现，有截图或数据证明 |
| 12 | 提供 3-4 个具体案例 | 信息较完整，有数据支撑 |
| 8 | 提供 1-2 个案例 | 信息不够详细，仅有口头描述 |
| 5 | 仅有笼统描述 | 无具体案例，如"用户反馈不好用" |
| 0 | 无法说明问题是否存在 | 完全无证据 |

**硬证据示例：**
- 客服工单截图（含时间、用户 ID、问题描述）
- 用户反馈埋点数据（NPS、满意度评分）
- 业务数据异常（转化率下降、流失率上升）
- 操作日志（用户操作失败记录）

### 2. 影响范围（10 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 10 | 影响 100+ 人/天或核心业务 | 有精确数据，如 DAU/MAU、业务报表 |
| 8 | 影响 50-100 人/天或重要业务 | 有数据估算，逻辑合理 |
| 5 | 影响 10-50 人/天或非核心业务 | 有定性描述，无精确数据 |
| 2 | 影响 <10 人/天且非核心业务 | 仅有业务方声称 |
| 0 | 无法说明影响范围 | 完全无证据 |

**硬证据示例：**
- DAU/MAU 数据报表
- 业务流程涉及人数统计
- 财务数据（影响收入金额）
- 操作频率统计

### 3. 当前应对（5 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 5 | 清晰说明现有方案及不足 | 有流程文档、操作日志、时间消耗统计 |
| 3 | 说明现有方案但不够详细 | 有口头描述，无数据支撑 |
| 1 | 仅说明有现有方案 | 无具体说明 |
| 0 | 未说明或无现有方案 | - |

## 二、价值性评估（40 分）

### 1. 收益可量化（20 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 20 | 收益完全量化，测算逻辑清晰 | 有收入/效率/体验提升的具体数字和测算公式 |
| 15 | 主要收益可量化 | 有主要指标量化，测算逻辑基本合理 |
| 10 | 部分收益可量化 | 有部分指标，测算逻辑有瑕疵 |
| 5 | 仅有定性描述 | 如"提升用户体验"、"增加用户粘性" |
| 0 | 无法说明任何收益 | - |

**硬证据示例：**
- 收入测算表（预期增收金额、计算公式）
- 效率提升计算（节省工时、人力成本）
- A/B 测试预估（基于历史数据）
- 竞品数据参考（有来源链接）

### 2. 投入产出比（10 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 10 | ROI ≥ 3:1，测算清晰 | 有开发工时估算、资源成本核算 |
| 7 | ROI 1:1 ~ 3:1 | 有粗略估算 |
| 4 | ROI < 1:1 但有战略价值 | 有说明 |
| 0 | 未考虑或无法评估 | - |

**硬证据示例：**
- 开发工时估算（技术评审记录）
- 资源成本核算（人力、服务器、第三方服务）
- ROI 计算过程

### 3. 战略对齐（10 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 10 | 完全对齐当前 OKR/产品方向 | 有 OKR 文档引用、产品路线图 |
| 7 | 部分对齐 | 有相关性说明 |
| 4 | 不直接相关但无冲突 | - |
| 0 | 与战略方向冲突 | - |

**硬证据示例：**
- 当前季度 OKR 文档
- 产品路线图
- 业务优先级排序

## 三、可行性评估（30 分）

### 1. 技术可行性（10 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 10 | 技术无难点，可直接开发 | 有技术方案或技术评审记录 |
| 7 | 有难点但有解决方案 | 有技术调研、方案对比 |
| 4 | 有难点，方案不确定 | 需要技术预研 |
| 0 | 技术不可行或有重大依赖 | - |

**硬证据示例：**
- 技术方案文档
- 架构评审记录
- 第三方 API 文档
- 技术调研报告

### 2. 资源匹配（10 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 10 | 当前人力/时间完全满足 | 有排期表、人力日历 |
| 7 | 需要调整但可满足 | 有调整方案 |
| 4 | 资源紧张，需要协调 | 有协调计划 |
| 0 | 资源完全不匹配 | - |

**硬证据示例：**
- 项目排期表
- 团队人力日历
- 资源分配表

### 3. 优先级（10 分）

| 得分 | 标准 | 硬证据要求 |
|:----:|------|------------|
| 10 | P0 紧急重要 | 有需求池排序、紧急程度标签 |
| 7 | P1 重要不紧急 | - |
| 4 | P2 不重要但紧急 | - |
| 0 | P3 不重要不紧急 | - |

**硬证据示例：**
- 需求池排序
- 紧急程度标签
- 业务方优先级声明

## 证据可信度加分

| 证据类型 | 加分规则 |
|----------|----------|
| 多个独立硬证据交叉验证 | +5 分（单项上限） |
| 有 A/B 测试或历史数据支撑 | +3 分 |
| 有第三方报告或行业 benchmark | +2 分 |

## 证据可信度扣分

| 情况 | 扣分规则 |
|------|----------|
| 仅凭业务方口头声明 | -3 分 |
| 数据来源不明确 | -2 分 |
| 测算逻辑有明显漏洞 | -3 分 |
