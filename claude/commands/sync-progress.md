# Claude Command: Sync Progress

同步当前会话进度到所有相关文件，并进行全量一致性检查。

---

## ⚠️ 前置条件：检查历史错误（必须执行）

> **这个错误已经犯了 N+1 次！每次执行前必须先检查！**

**执行步骤**:
1. **识别当前项目并读取 MEMORY.md 中的错误速查表**
   - 根据当前工作目录确定项目标识符（如 `d:\projects\my-project` → `d--projects--my-project`）
   - 文件路径: `~/.claude/projects/{项目标识符}/memory/MEMORY.md`
   - 重点关注 "常见错误速查" 章节

2. **检查相关错误记录**
   | 关键词 | 检查项 |
   |--------|--------|
   | sync-progress | 是否遗漏更新文件？是否跳过步骤未说明？ |
   | 核对 | 是否实际读取源文件对比？是否只看了部分文档？ |
   | 文档 | 是否核对代码示例与实际代码一致？ |

3. **确认不会重复犯错**
   - 如果有相关错误记录，必须明确说"已检查错误 X，本次不会重复"

**输出示例**:
```
📋 错误检查（前置条件）
- [x] 已读取 MEMORY.md 错误速查表
- [x] 相关错误：sync-progress 只更新项目文档（03-18）
- [x] 本次不会重复：将更新所有文件并检查一致性
```

---

## ⚠️ 会话开始时的路径约定

**用户应在会话开始时指定需求文档路径**，格式如：

```
"需求文档在 D:\docs\requirements.md"
```

**系统将自动推断以下路径**：

| 文件类型 | 推断规则 | 示例 |
|---------|---------|------|
| 技术文档 | 同目录 + "技术文档.md" | `D:\...\technical-doc.md` |
| 改动说明 | 同目录 + "改动说明.md" 或 "代码改动说明.md" | `D:\...\changelog.md` |
| OpenSpec变更 | 根据"my-feature"或需求关键词匹配 | `openspec/changes/my-feature-v101/` |

**如果未在会话开始时指定路径**：

```
⚠️ 未找到需求文档路径

请提供需求文档路径，我将自动推断技术文档和改动说明的位置：
例如："D:\docs\requirements.md"

或者分别提供：
1. 技术文档路径
2. 改动说明路径
3. OpenSpec变更ID（如：my-feature-v101）
```

---

## Usage

```
/sync-progress            # 全量同步 + 一致性检查（推荐）
/sync-progress verify     # 仅检查一致性，不更新文件
/sync-progress planning   # 只更新 planning-with-files
/sync-progress openspec   # 只更新 openspec
/sync-progress docs       # 只更新项目文档
/sync-progress all        # 更新所有文件（同默认）
```

---

## 全量检查范围

执行 `/sync-progress` 时，将检查以下所有文件的一致性：

### 1. Planning-with-files

| 文件 | 检查内容 |
|------|---------|
| `task_plan.md` | Current Phase 是否与实际进度一致 |
| `progress.md` | 最近会话记录是否与实际工作一致 |
| `findings.md` | 技术决策是否与代码实现一致 |

### 2. OpenSpec 变更

| 文件 | 检查内容 |
|------|---------|
| `openspec/changes/{变更ID}/tasks.md` | 任务状态是否与代码完成情况一致 |
| `openspec/changes/{变更ID}/spec.md` | 规格描述是否与实现一致 |

### 3. 项目文档

| 文件 | 检查内容 |
|------|---------|
| 技术文档 | 代码示例是否与实际代码一致 |
| 改动说明 | 文件清单是否与实际变更一致 |

## Workflow

### 阶段 0: 路径确认（必做）

**从会话上下文获取需求文档路径**：

```
步骤1: 检查会话上下文是否有需求文档路径
步骤2: 如有，推断技术文档和改动说明路径
步骤3: 如无，询问用户提供路径
```

**路径推断逻辑**：

```python
# 假设需求文档路径为：
需求文档 = "D:\docs\requirements.md"

# 推断技术文档：
目录 = "D:\docs\"
技术文档 = 目录 + 同前缀 + "技术文档.md"
# 例如：technical-doc.md

# 推断改动说明：
改动说明 = 目录 + "changelog.md" 或 类似命名
# 通过 glob 目录下 *改动说明*.md 或 *代码改动*.md 查找

# 推断 OpenSpec 变更：
变更ID = 根据"my-feature"匹配 openspec/changes/ 下的目录
# 例如：my-feature-v101
```

**输出**：
```
📋 路径确认
- 需求文档: D:\...\requirements.md
- 技术文档: D:\...\technical-doc.md ✓
- 改动说明: D:\...\changelog.md ✓
- OpenSpec: openspec/changes/my-feature-v101/ ✓
```

---

### 阶段 1: 全量一致性检查（必做）

> **不限于当前会话，检查所有文件的一致性。**

#### 1.1 检查 planning-with-files

```
检查 task_plan.md:
  - Current Phase 状态是否正确？
  - Decisions Made 是否需要更新？

检查 progress.md:
  - 最近会话的 Files Modified 是否与实际一致？
  - 是否有遗漏的工作记录？

检查 findings.md:
  - Technical Decisions 是否与代码实现一致？
```

#### 1.2 检查 openspec 变更

```
检查 tasks.md:
  - 任务完成状态是否与代码一致？
  - 是否有任务已完成但未标记？

检查 spec.md（如存在）:
  - 接口描述是否与实现一致？
```

#### 1.3 检查项目文档（核心步骤）

**强制执行流程**:

```
步骤1: 扫描技术文档中的所有代码示例
       - 找出所有 ```java 代码块
       - 标注每个代码块对应的源文件路径

步骤2: 逐个读取实际源文件
       - 用 Read 工具读取每个对应的 .java 文件
       - **不是回忆**, **不是假设**, 是实际读取

步骤3: 逐项对比
       - 方法签名（参数类型、返回类型）
       - 变量名、常量名
       - 判断条件
       - 执行顺序
       - 异常处理

步骤4: 记录所有不一致

步骤5: 更新文档 + 添加修订记录
```

**自检清单**（执行本步骤时必填）:
- [ ] 已扫描技术文档中所有代码示例
- [ ] 已读取所有代码示例对应的源文件
- [ ] 已逐项对比方法签名、返回值、判断条件
- [ ] 发现的不一致已全部记录
- [ ] 修订记录中已列出修正内容

---

### 阶段 2: 分析当前会话增量

**对比历史记录与当前状态**：

```
历史进度（来自 progress.md）:
- 上次会话: 2026-03-18 敏感字段修改
- 已完成文件: [列表]

当前会话增量:
- 新完成的工作: [列表]
- 新修改的文件: [列表]
- 新做的决策: [列表]
```

---

### 阶段 3: 更新所有文件

#### 3.1 更新 planning-with-files

**task_plan.md:**
- 更新 Current Phase 状态
- 记录 Decisions Made
- 记录 Errors Encountered

**progress.md:**
- 添加当前会话的 Actions Taken
- 记录 Files Created/Modified
- 记录 Test Results
- 更新 Pending Items

**findings.md:**
- 记录 Research Findings
- 记录 Technical Decisions
- 记录 Issues Encountered

#### 3.2 更新 openspec

**tasks.md:**
- 更新任务完成状态 `[ ]` → `[x]`
- 刷新更新时间戳

#### 3.3 更新项目文档

**技术文档:**
- 更新代码示例（如有不一致）
- 添加修订记录
- 更新架构决策

**改动说明:**
- 更新文件清单
- 更新接口说明

## 特殊模式

### `/sync-progress verify` - 仅检查模式

只执行一致性检查，不更新任何文件：

```
用途: 在提交代码前检查文档是否需要更新
输出: 列出所有不一致的地方，由用户决定是否修复
```

---

## When to Use

- 会话结束前（推荐）
- 完成重要里程碑后
- 提交代码前（使用 verify 模式）
- 长时间中断后恢复工作时（检查是否有遗漏更新）

## Notes

- **全量检查**: 不限于当前会话，检查所有相关文件
- **路径推断**: 根据需求文档路径自动推断其他文档位置
- **智能修正**: 发现不一致自动更新文档
- **修订记录**: 每次更新都记录修订内容
