# AI Workflow Control Kit — 深度分析（更新版）

> 分享准备材料。所有数字均经仓库源码核查，可复现命令见附录。

---

## 一、TL;DR

**它是一个可迁移的 AI 编码"控制平面"：用技能路由、生命周期钩子和无人值守回放评测，把 AI 编码从"凭感觉对话"约束成可安装、可审查、可验证、可回放的工程流程。**

它不是 AI agent、不是模型、不是 IDE 插件——是在 Claude Code / Codex 等 AI 编码宿主外层加装的**控制层**，把工作流规则、触发条件、验证门禁和评估机制提前固化。

**成熟度判断（2026-06-14）：** 当前仓库已经形成可安装、可校验、可迁移的 v1 控制平面；`workflow-history` 已更新到 v007，`replay-autopilot` 已同步到知识库 changelog v531 / 回归 v565 的控制面源码。需要如实说明：replay 目录仍保留当前 benchmark 的 `configs/`、`requirements/` 和部分回归 fixture，用于验证控制平面，不应表述为“所有 replay 材料零业务名残留”。最近已知回放 r52 在 Phase 1 前被 Plan 门禁拦截，v531 修复的是该 Plan 门禁误拦截，并不等同于已经完成下一轮端到端交付。

## 二、是什么 / 不是什么

| 是 | 不是 |
|----|------|
| AI 编码宿主外层的控制平面 | AI agent / 模型 / IDE 插件 |
| 可安装、可校验的技能+hook+规则包 | 某个业务项目的实现 |
| 自包含的回放评测引擎 | 运行态会话或私密状态 |
| 个人工程化 → 产品化的真实过渡样本 | 已完成的产品 |

**仓库有意不包含：** token、API key、会话日志、SQLite 状态、业务源码、oracle diff、`.env`。

---

## 三、为什么需要它

| 痛点 | 表现 |
|------|------|
| AI 对话零标准 | 每次会话靠临时对话定规则，回到"我该先干嘛" |
| 审查质量不稳定 | AI 提交质量依赖个体发挥，没有固定门禁 |
| 重做不知原因 | 事后无法还原"那次为什么走了那条路" |
| 多机器同步贵 | 换电脑要人工回忆技能/hook/配置写在哪 |
| 工作流无法演进 | 改了技能后没有评估手段，不知道改得好不好 |

核心思路：**把"应该怎么做"写成可复用控制规则，而不是每次靠 AI 自行判断。**

## 四、怎么做到的：四层控制

### 4.1 路由层 — workflow-router（入口技能）

决定所有请求分流，只做分流和门禁提示，不内联执行下游技能（Iron Law）。

**六字段冻结工作形态：** `MODE → SCOPE → CONTRACT → VERIFY → EXIT → SYNC`

**五种 Mode：**

| Mode | 场景 | 默认动作 |
|------|------|----------|
| `planned-dev` | 需求/设计/功能实现/多 surface | 需求冻结→设计→审批→TDD→收口 |
| `hotfix` | 有失败测试/堆栈/复现的小修 | 锁证据→RED→最小 diff→回归 |
| `debug-only` | 只排查/查询/给结论 | 证据锚点→假设表→信息缺口 |
| `review-only` | 只审查/比较 | findings→风险→验证建议 |
| `tooling` | 技能/hook/配置维护 | canonical source→备份→smoke→回滚 |

**Size 分级：** `quick` / `small`(≤2文件) / `medium`(3-10) / `large`(>10，跨服务/DB/外部)。

**正常开发推荐链：**
```
pre-flight-check → req-alignment-check → ideate:planning-brainstorm
  → deep-plan（必须同时出 .doc/ 和 openspec/）
  → [用户审批，唯一暂停点]
  → dev-workflow → gen-tests → deep-review → sync-progress
```

### 4.2 实现层 — dev-workflow（最核心技能）

**8 个 Phase（编号 0-7，3.5 可选），每个 Phase 必须有产出才能进下一个。**

| Phase | 产出 |
|-------|------|
| 0 | MODE/SCOPE/CONTRACT/VERIFY/EXIT/SYNC 冻结表 |
| 1 | 隐藏需求清单 |
| 2 | `.doc/` + `openspec/` + Test Design Control（缺一不可）|
| 3 | 用户审批（唯一暂停点，BLOCKING）|
| 3.5 | 工时评估（可选）|
| 4 | 代码 + tasks.md（按 core_path → supporting_surface）|
| 5 | gen-tests（必须消费 Test Design Control）|
| 6 | 注释（条件性）|
| 7 | Handoff + `.doc`/`openspec`/`.memory` 三路同步 |

**Hard Gates（SKILL.md 明确）：**
- Phase 3 审批不可跳过
- Phase 2 必须同时维护 `.doc/` 与 `openspec/`
- 实现遇新约束（DTO/接口/配置/扩展性）必须回退 Phase 2，不补丁式推进
- 无可运行反馈回路时不准靠读代码修 bug
- 收口缺三路同步账本只能 `PARTIAL`

### 4.3 Hook 层 — Node 生命周期链

5 个核心 Node hook（4 个在 `agents/hooks/` + 1 个 codex `skill-tracker.js`）：

| 事件 | 脚本 | 作用 |
|------|------|------|
| `UserPromptSubmit` | `skill-activation-prompt.js` | 匹配 `skill-rules.json` 关键词→输出技能建议 |
| `Stop` | `skill-execution-receipt.js` | 回看 transcript→识别实际用技能→对比编辑文件→落盘回执 |
| `FileChanged` | `workflow-sync-state.js` | 记录编辑分类和时间戳 |
| `SessionStart` | `workflow-sync-state.js` | 恢复/刷新 per-project 状态 |
| `Stop`(Codex) | `skill-tracker.js` | 追踪 Codex 技能调用 |

**为什么用 Node 而非 PowerShell：** Windows PowerShell 5.1 在高频 hook（尤其 `UserPromptSubmit`）触发 R6016 崩溃（`not enough space for thread data`）。Node 直接 `execFile` 不经 shell，更稳定。这是真实踩坑——`workflow-history/v003`、`v004` 都是这个问题的迭代。

### 4.4 Replay 评测层 — replay-autopilot

仓库最独特的部分：**把 AI 编码任务重复执行，用脚本化闭环评分判断技能改动是否有效。**

**四层 Pipeline：**
```
Phase 0（合同门禁）→ Phase 0.5（规划竞争择优）
  → Phase 1（逐 Slice 执行，Run-SliceLoop）→ Phase 2（Oracle 后验评分）
```

**熔断：** 覆盖率达标 / 无提升轮次 / 零覆盖 / Stop-Loss（多条件）。

**保护：** 隔离 worktree / Source-of-Truth Gate / Carrier Authorization / fail-closed 默认 / Exact Contract Normalizer。

**反哺：** replay 失败蒸馏成 `skills-manifest.md` 的 "Replay Lessons Integrated"——**实测 132 条**控制规则。典型：`non-failing RED is zero executable delta`、`mocked return values cannot count as GREEN`。

## 五、设计哲学

`skills-manifest.md` 的 "Workflow North Star" 列出 **13 条**不可商量原则。精选：

1. **需求覆盖 > 代码体量** — 看需求命中率，不看 diff 行数
2. **核心主链 > 辅助片段** — 先闭环真实入口
3. **可执行证据 > 静态自信** — 测过/跑过/看过日志才算数
4. **验证封顶覆盖 > 自我报告** — 覆盖率由 verifier 定
5. **宿主中性** — CLI/会话/hooks 放适配层，基础层不感知
6. **项目教训进基础层前必须蒸馏** — 不含项目名/类名/表名
7. **暂存分类精细** — `git add .` 永远不是默认

## 六、工程深度：为什么它不是 Demo

**6 层架构：** 路由 → 技能 → hook → 宿主适配 → replay → workflow-history。

**可核查的工程量：**

| 维度 | 数量 | 核查命令 |
|------|------|----------|
| 技能 | **33** 目录（skill-rules.json 注册 31）| `find agents/skills -maxdepth 1 -mindepth 1 -type d \| wc -l` |
| Replay lessons | **132** 条 | `sed -n '/## Replay Lessons Integrated/,$p' skills-manifest.md \| grep -cE '^[0-9]+\.'` |
| replay-autopilot 脚本 | **366** 个 PowerShell 脚本（含 **251** 个 `Test-v*.ps1` 回归，最高 v565）| `Get-ChildItem -Recurse replay-autopilot -Filter "*.ps1"` |
| 核心 Node hook | **5** | `find agents/hooks codex/hooks/scripts -name "*.js"` |
| North Star 原则 | **13** | `grep -cE "^[0-9]+\." <North Star 段>` |

**演化轨迹（workflow-history v000-v007 + 知识库 changelog 至 v531）：** 每一步都是对真实问题的回应——bootstrap → 本地化变更索引 → 拆 hxld_vault 依赖 → 解决 PowerShell R6016 → 全面 Node-first → 架构文档 → 选择性 RTK 诊断 → 同步最新 replay 控制面并补充客观演化记录。知识库 changelog 显示 replay-autopilot 又继续演进出 Plan 合同、carrier/oracle 绑定、TDD 证据门禁、stopline、executor 资源/认证门禁、read-only 分类、reuse 清理和 v531 Plan harness gate 修复。

Demo 不会产出 132 条踩坑控制信号，不会有 `mock_behavior_gap` → `exact_contract_gap` 这种精确 gap flag 体系，更不会有针对 R6016 的迭代记录。

> **客观补充：** 132 条 lesson 全是作者本人 replay 产出，无第三方复现。它证明工程深度真实，但不能单独证明对别人也有效。

## 七、产品化成熟度与当前边界

对照 `docs/PRODUCTIZATION_GUIDE.md` 的三阶段路线图，当前状态更准确的判断是：

| 阶段 | 当前判断 | 依据 |
|------|----------|------|
| Stage 1 — 可迁移 | 基本完成 | Node-first 安装/校验/安全扫描脚本齐备；`workflow-history` 仓库内自包含；默认安装目标来自用户目录或显式参数 |
| Stage 2 — 参数化控制平面 | 主体完成，需注明 replay benchmark 边界 | kit 层和安装层已参数化；但同步最新 replay 后，`configs/`、`requirements/` 和部分测试 fixture 保留 benchmark 术语，用于回归评测 |
| Stage 3 — 可评测工作流运行时 | 控制面能力持续增强 | replay 控制器已有 Phase gate、stop-and-evolve、failure audit、hard reflection、TDD/charter/evidence 门禁，回归最高到 v565 |

可交差的客观事实：

- 仓库本身已经是一个可安装、可校验、可迁移的 AI workflow control kit。
- `replay-autopilot` 已同步到知识库 changelog v531 对应控制面，包含 251 个 `Test-v*.ps1` 回归。
- 变更历史已经从仓库 bootstrap 记录到 v007，并在 v007 中补充从早期 kit 到 v531 控制面的演化摘要。
- 不包含 token、运行态会话、SQLite、日志、私密 oracle diff 或业务源码。

需要避免的过度表述：

- 不说“全部 replay 材料零业务命名耦合”。最新 replay 保留 benchmark case 作为评测输入，这是事实。
- 不说“最新版本已经完成端到端自主交付”。已知 r52 止步于 Phase 1 前，v531 修复的是 Plan 门禁误判。
- 不把 `rdc-git` 这类公司特定技能说成通用能力；它是可插拔公司流程层。
